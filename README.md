# Pi Agent Configuration

This repository is managed with [yadm](https://yadm.io/) and tracks my portable Pi coding-agent configuration in place, without duplicating files.

## What is tracked

- `.pi/agent/settings.json`
- `.pi/agent/models.json`
- `.pi/agent/AGENTS.md`
- `.pi/agent/extensions/`
- `.local/bin/pi-codegraph-hooks` for optional repository-local CodeGraph Git hooks
- `.pi/prompts/opsx-*.md` generated by OpenSpec for Pi slash prompts
- `.agents/skills/` for portable custom skills that are not managed by Codex/OpenSpec, including:
  - `codegraph-cli`
  - `code-reviewer`, `docs-writer`, `pr-creator`
  - `godot-cli`, `openscad-cli`
  - `skill-creator`, `skill-installer`
- Documentation for optional Context Mode and RTK workflow setup. Context Mode's npm package is configured for Pi; generated local databases and caches are not tracked. RTK itself and any generated user-level hooks are not tracked.
- Optional Pi resources if added later: `keybindings.json`, `AGENTS.md`, `SYSTEM.md`, `APPEND_SYSTEM.md`, `prompts/`, `themes/`

## What is intentionally not tracked

- `.pi/agent/auth.json` — credentials and OAuth tokens
- `.pi/agent/sessions/` — conversation history and potentially sensitive project context
- `.pi/agent/run-history.jsonl`
- `.pi/agent/update-cache.json`
- `node_modules/` and other generated caches
- `.codegraph/` — generated CodeGraph indexes; keep them project-local unless a repo explicitly chooses otherwise

## Install yadm

### Windows

Use Git Bash or another Bash-compatible shell. Install yadm into `~/.local/bin`:

```bash
mkdir -p ~/.local/bin
curl -L --fail -o ~/.local/bin/yadm https://github.com/yadm-dev/yadm/raw/master/yadm
chmod +x ~/.local/bin/yadm
yadm --version
```

Make sure `~/.local/bin` is on your `PATH`.

If you use WSL, install yadm from your Linux distribution instead, for example:

```bash
sudo apt update
sudo apt install yadm
```

### Linux

Debian/Ubuntu:

```bash
sudo apt update
sudo apt install yadm
```

Fedora:

```bash
sudo dnf install yadm
```

Arch Linux:

```bash
sudo pacman -S yadm
```

### macOS

Using Homebrew:

```bash
brew install yadm
```

## Restore on a new machine

Install yadm, then clone this repo into your home directory:

```bash
yadm clone git@github.com:dobleuber/pi-agent.git
```

Then authenticate providers again:

```bash
pi
/login
```

## Install the currently used skills

Pi loads skills from three places in this setup:

```text
~/.agents/skills/   # custom portable skills tracked by this repo
~/.codex/skills/    # Codex-managed skills
~/.pi/skills/       # OpenSpec-generated Pi skills
```

`.pi/agent/settings.json` keeps `~/.codex/skills` and `~/.pi/skills` enabled so future skills installed there are picked up. Only known duplicates that also exist in `~/.agents/skills` are excluded from the Codex path.

Install/restore them with:

### 2. CodeGraph CLI workflow

Source: <https://github.com/colbymchenry/codegraph>

Pi does not use CodeGraph through MCP in this setup. Instead, Pi uses the globally available `codegraph` CLI and the tracked global instructions in `~/.pi/agent/AGENTS.md` plus the `~/.agents/skills/codegraph-cli` skill.

Install CodeGraph globally:

```bash
npm install -g @colbymchenry/codegraph
```

Update CodeGraph later with the same command:

```bash
npm install -g @colbymchenry/codegraph@latest
```

Verify:

```bash
codegraph --version
```

Expected Pi workflow in code repositories:

- Before broad repository exploration with `rg`, `find`, or multi-file reads, Pi should apply the `codegraph-first-exploration` guardrail skill when available. That skill exists to stop Pi from falling back to manual search before trying CodeGraph; `codegraph-cli` remains the detailed command reference.

- Before significant code exploration, architecture analysis, debugging, refactoring, or impact analysis, if `.codegraph/` is missing, Pi should initialize the repo:

  ```bash
  codegraph init -i
  ```

- When `.codegraph/` exists, Pi should prefer CodeGraph before broad grep/read exploration:

  ```bash
  codegraph context "<task>"
  codegraph query "<symbol>"
  codegraph callers "<symbol>"
  codegraph callees "<symbol>"
  codegraph impact "<symbol>"
  ```

- Before relying on CodeGraph when the index may be stale, especially after branch changes, merges, rebases, significant edits, debugging, refactoring, or impact analysis, refresh it:

  ```bash
  codegraph sync
  ```

Treat `.codegraph/` as generated local state. Add it to each project's ignore rules unless that project explicitly documents a different policy.

#### Optional CodeGraph Git hooks per repository

Use the tracked helper to install repository-local hooks in selected repos. These are opt-in; this setup does not install global Git hooks.

From inside a target repository:

```bash
pi-codegraph-hooks install
```

This installs defensive `post-checkout`, `post-merge`, and `post-rewrite` hook blocks. They no-op when `codegraph` is missing or `.codegraph/` does not exist, and otherwise run `codegraph sync` after branch/history changes.

To also print non-blocking affected-test information during commits:

```bash
pi-codegraph-hooks install --pre-commit
```

Remove managed hook blocks:

```bash
pi-codegraph-hooks remove
```

### 3. Context Mode context-hygiene workflow

Source: <https://github.com/mksglu/context-mode>

Context Mode is configured as a Pi package to add context-hygiene tools and session-continuity hooks. It complements CodeGraph rather than replacing it: CodeGraph remains the first choice for code architecture, symbol lookup, callers/callees, and impact analysis; Context Mode is for large outputs, logs, fetched content, bulk data analysis, and avoiding raw data dumps into the model context.

Install or update the global CLI and Pi package:

```bash
npm install -g context-mode
pi install npm:context-mode
```

`.pi/agent/settings.json` includes the package entry:

```json
"npm:context-mode"
```

Restart Pi, or reload resources from inside Pi when appropriate:

```text
/reload
```

Verify the installation:

```bash
context-mode --help
npm list -g context-mode --depth=0
```

Then, inside a fresh Pi session, verify the registered Context Mode tools with:

```text
ctx stats
```

Expected routing hierarchy:

- Use CodeGraph first for repository architecture, symbols, call flow, and impact analysis.
- Use Context Mode `ctx_*` tools for large or noisy outputs, logs, web/fetched content, generated analysis, and session-continuity queries.
- Use Pi's direct `read` tool for exact source-file inspection.
- Use Pi's `edit`/`write` tools for precise changes.

Context Mode stores generated session/content data outside this yadm repository; do not commit local Context Mode databases, caches, or generated runtime state.

### 4. RTK token-saving workflow (optional)

Source: <https://github.com/rtk-ai/rtk>

RTK is an optional CLI proxy that filters and compresses common development command output before it reaches an AI-agent context. This can reduce token usage during Pi/Codex/OpenSpec sessions, but the setup remains fully usable without RTK.

Install RTK using one of the upstream-supported methods:

```bash
# Linux/macOS quick install
curl -fsSL https://raw.githubusercontent.com/rtk-ai/rtk/refs/heads/master/install.sh | sh

# Or, with Cargo from the Git repository
cargo install --git https://github.com/rtk-ai/rtk
```

Homebrew users can also install it with:

```bash
brew install rtk
```

Initialize RTK for the AI agents you use. For this Pi-oriented setup, start with the generic global hooks; add Codex-specific hooks if you also use Codex directly:

```bash
rtk init -g            # Claude Code / Copilot-style generic hooks
rtk init -g --codex    # Codex CLI hooks
```

RTK also supports other agents upstream, for example:

```bash
rtk init -g --gemini
rtk init -g --agent cursor
```

Restart the affected agent after initialization. RTK-generated hook/config files are user-level machine state; do not add them to this yadm repository unless they are reviewed later and proven safe, stable, and portable.

Verify:

```bash
rtk --version
rtk gain
rtk git status
```

Expected workflow:

- For supported Bash tool commands after initialization and restart, RTK can automatically rewrite/proxy commands such as `git status` and return compact output.
- Agent built-in tools such as file readers, grep/search tools, or structured read tools do not necessarily pass through Bash hooks. When compact output matters, run explicit RTK commands instead:

  ```bash
  rtk git status
  rtk read README.md
  rtk grep "openspec" .
  rtk test "npm test"
  ```

- When debugging requires full raw output, bypass RTK by running the original command directly or by using the relevant non-RTK agent tool.

RTK is not required for Pi, OpenSpec, or any tracked skills. If RTK is absent or removed, the existing workflows continue to operate normally.

### 5. Varlock secret-management workflow (optional)

Source: <https://varlock.dev/> / <https://github.com/dmno-dev/varlock>

Varlock is an optional secret-management and `.env` workflow for AI-assisted development. It lets agents inspect schema and configuration context without seeing secret values, validates environment variables before commands run, can encrypt local overrides, and can scan for leaked secrets. This setup remains fully usable without Varlock.

Install Varlock using one of the upstream-supported methods:

```bash
# For JavaScript projects, initialize through npx from the project root
npx varlock init

# Agent/automation-friendly initialization
npx varlock init --agent

# Or install the standalone binary with Homebrew
brew install dmno-dev/tap/varlock

# Or install the standalone binary with the upstream installer
curl -sSfL https://varlock.dev/install.sh | sh -s
```

Verify the CLI and consult upstream help for the installed version:

```bash
varlock --version
varlock --help
varlock init --help
varlock load --help
varlock run --help
varlock scan --help
```

If Varlock is only used through `npx`, replace `varlock` with `npx varlock` in the commands above.

Recommended Pi workflow:

- Use `.env.schema` for agent-readable configuration context. It can describe required variables, types, docs, and sensitivity markers without containing secret values.
- Keep secret values in local, ignored files such as `.env.local`, or load them from an upstream-supported secret provider. Do not commit plaintext secret values.
- Use `varlock load --agent` when an agent needs a redacted, safe summary of available configuration.
- Use `varlock run -- <command>` to run a process with resolved and validated environment variables injected.
- For interactive AI-agent CLIs, prefer disabling stdout redaction and graph injection unless you specifically need them:

  ```bash
  varlock run --no-redact-stdout --no-inject-graph -- pi
  ```

Example schema with placeholders only:

```dotenv
# .env.schema - safe for agents to read
# @sensitive @required @type = string(startsWith = sk-ant-)
# @docs (https://docs.anthropic.com/)
ANTHROPIC_API_KEY=

# @sensitive @required @type = string
OPENAI_API_KEY=

# @type = enum(development, preview, production, test)
APP_ENV=development
```

Example local override using dummy values only:

```dotenv
# .env.local - keep ignored and local to this machine
ANTHROPIC_API_KEY=sk-ant-placeholder-do-not-use
OPENAI_API_KEY=sk-placeholder-do-not-use
```

Encrypt local sensitive values in place when appropriate:

```bash
varlock encrypt --file .env.local
```

Inspect and load configuration safely:

```bash
varlock load --agent                 # redacted agent-safe summary
varlock load --format shell          # shell export format
varlock reveal ANTHROPIC_API_KEY     # secure human reveal, avoid pasting into prompts
varlock printenv APP_ENV             # print a single non-sensitive value when needed
```

Run common commands with Varlock-managed environment variables:

```bash
varlock run -- npm test
varlock run -- python script.py
varlock run --no-redact-stdout --no-inject-graph -- pi
```

Scan for leaks before committing:

```bash
varlock scan
varlock scan --staged
# Optional per-repository hook managed by Varlock itself:
varlock scan --install-hook
```

Secret-safety rules:

- Never commit real API keys, provider tokens, registry tokens, deployment credentials, `.env.local`, generated local Varlock state, or machine-specific secret configuration.
- Keep examples limited to placeholder names and dummy values.
- Prefer `varlock load --agent` for agent-readable summaries instead of exposing full environment dumps.
- Use upstream `varlock <command> --help` and <https://varlock.dev/> as the authoritative command reference when behavior differs from these notes.

Varlock is not required for Pi, OpenSpec, CodeGraph, RTK, or any tracked skills. If Varlock is absent or removed, the existing workflows continue to operate normally.

Current CLI version used when these skills were generated:

```bash
openspec --version

```

Install OpenSpec:

```bash
npm install -g @fission-ai/openspec@latest
# For an exact match with this machine instead:
# npm install -g @fission-ai/openspec@1.2.0
```

This setup uses OpenSpec's **custom expanded workflow** with Pi output. Configure OpenSpec:

```bash
openspec config profile
```

Choose:

```text
profile: custom
delivery: both
workflows:
  explore
  new
  continue
  apply
  ff
  sync
  archive
  bulk-archive
  verify
  onboard
```

OpenSpec's Pi integration generates skills into `~/.pi/skills` and prompts into `~/.pi/prompts`. Keep those locations; Pi is configured to load both.

```bash
cd ~

openspec update --force
```
